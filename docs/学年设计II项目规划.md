# 重庆市二手房源价格数据分析与可视化项目规划

> 规划依据：
> - 《2024级信管+大数据学年设计II指导手册》：明确主题、功能要求、非功能要求、提交材料和答辩要求。
> - 《学年设计II文档模板》：明确最终说明书章节结构与封面信息。
> - 当前代码库：现有工程是一个默认 Android Kotlin 项目，尚未包含 Ktor 后端模块。

## 1. 项目目标

本项目以“重庆市二手房源价格数据分析与可视化”为主题，使用 Kotlin 技术栈完成一个可演示、可说明、可提交的系统：

1. 使用 Ktor Server 搭建后端，完成房源数据采集、清洗、存储、增量更新、统计分析和 API 输出。
2. 使用 Android Kotlin 作为前端 App，调用后端 API 展示房价概览、区域对比、筛选列表、可视化图表和分析结论。
3. 数据规模目标不少于 5 万条；如果实际网站采集受限，保留“合规采集 + 本地缓存 + 可复现导入”的兜底方案。
4. 最终按模板补全《学年设计II文档》，正文不少于 8000 字，并包含系统设计、实现过程、关键代码、运行截图和结论。

## 3. 技术路线

### 3.1 工程结构

计划在现有 Android 工程基础上扩展为多模块 Gradle 项目：

```text
Houses
├─ app        Android 前端 App，展示数据和图表
├─ server     Ktor 后端，提供采集、清洗、存储、分析和 API
└─ shared     可选共享模块，放 DTO、枚举、请求/响应模型
```

优先方案是新增 `:server` 和 `:shared`。如果时间压力较大，可以先只新增 `:server`，前后端各自维护 DTO，后续再抽出共享模块。

### 3.2 后端 Ktor 设计

后端采用 Ktor Server，核心能力如下：

| 模块 | 职责 |
| --- | --- |
| `plugins` | 配置序列化、CORS、日志、状态页、路由 |
| `crawler` | 按区县/分页采集房源列表与详情，支持限速和失败重试 |
| `cleaner` | 清洗价格、面积、户型、楼层、年份、区县等字段 |
| `repository` | 数据库读写、去重、批次记录、查询聚合 |
| `analytics` | 区县统计、价格分布、KMeans 分群、结论生成 |
| `routes` | 对 Android App 暴露 REST API |

建议 API：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/health` | 后端健康检查 |
| `POST` | `/admin/crawl?mode=incremental` | 触发增量采集 |
| `GET` | `/api/listings` | 房源分页列表和筛选 |
| `GET` | `/api/stats/overview` | 总房源数、均价、中位数、最高/最低区县 |
| `GET` | `/api/stats/districts` | 各区县均价、数量、总价统计 |
| `GET` | `/api/stats/price-distribution` | 价格区间分布 |
| `GET` | `/api/stats/area-price` | 面积与单价散点数据 |
| `GET` | `/api/analysis/clusters` | KMeans 分群结果 |
| `GET` | `/api/analysis/conclusions` | 可直接展示/写入文档的分析结论 |

### 3.3 数据库与数据字段

MVP 阶段优先使用 SQLite 文件数据库，便于本机演示和提交；如果环境允许，再切换 PostgreSQL。数据库字段先围绕统计和可视化设计：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `source` | 数据来源 |
| `source_id` | 来源站点房源 ID |
| `title` | 房源标题 |
| `district` | 区县 |
| `community` | 小区 |
| `address` | 地址或片区 |
| `total_price_wan` | 总价，单位万元 |
| `unit_price_yuan_m2` | 单价，单位元/平方米 |
| `building_area_m2` | 建筑面积 |
| `rooms` / `halls` | 户型 |
| `floor` | 楼层信息 |
| `orientation` | 朝向 |
| `build_year` | 建成年份 |
| `tags` | 标签 |
| `detail_url_hash` | 详情页 URL 哈希，用于去重 |
| `first_seen_at` | 首次采集时间 |
| `last_seen_at` | 最近采集时间 |
| `batch_id` | 采集批次 |

清洗规则：

1. 去掉空价格、空面积、异常面积、异常单价记录。
2. 统一价格单位：总价转万元，单价转元/平方米。
3. 统一区县名称，如“渝北”“渝北区”归一为“渝北区”。
4. 户型提取为室、厅、卫等数值字段，便于筛选和算法处理。
5. 记录清洗前后的数量，用于文档中说明数据质量。

### 3.4 Android 前端设计

当前 `app` 模块还是默认页面，后续改造成数据可视化 App。MVP 页面如下：

| 页面 | 展示内容 |
| --- | --- |
| 总览页 | 房源总数、平均单价、中位单价、最高/最低均价区县 |
| 区县对比页 | 区县均价、房源数量、总价分布 |
| 房源列表页 | 按区县、价格、面积、户型筛选，展示分页列表 |
| 分析页 | KMeans 分群结果、典型区域画像、结论文本 |
| 关于/说明页 | 数据来源说明、更新时间、系统技术路线 |

前端调用 Ktor API，不直接访问数据库。图表优先使用 Android 可用图表库或自定义轻量图表，实际依赖以后续 Gradle 可解析为准。

## 4. 数据分析与挖掘方案

优先实现能稳定答辩、容易解释的分析方法：

1. 描述性统计：各区县均价、总价中位数、面积分布、房源数量排名。
2. 相关性分析：面积、总价、单价、建成年份之间的关系。
3. KMeans 分群：按单价、面积、总价、建成年份等特征把房源分为若干类，例如“核心高价小户型”“改善型中高总价”“外围低单价大面积”等。
4. 结论生成：后端根据统计结果生成结构化结论，前端展示，文档第 6 章引用。

可选增强：

1. 简单价格预测模型：用面积、区县、户型、年份估算单价或总价。
2. 增量更新对比：展示两次采集之间新增房源和均价变化。
3. 数据质量面板：展示清洗前数量、清洗后数量、去重数量、异常数量。

## 8. 风险与兜底

| 风险 | 影响 | 处理方案 |
| --- | --- | --- |
| 数据源反爬或网页结构变化 | 采集数量不足 | 限速、缓存、分批采集；保留 CSV/JSON 导入接口；记录采集策略，避免绕过登录或验证码 |
| 7 月 3 日前时间紧 | 中期展示不完整 | 先完成样例数据 + API + App 总览，采集和算法继续迭代 |
| Android 图表依赖冲突 | 前端图表延误 | 先用 RecyclerView/简单自定义 View 展示，后续再引入图表库 |
| Ktor/Gradle 版本不兼容 | 后端无法编译 | 优先匹配当前 Kotlin 2.0.21；必要时锁定兼容 Ktor 版本 |
| 文档与代码不同步 | 答辩容易被问住 | 每完成一个模块就记录截图、代码片段和实现说明，最后统一写入模板 |
