> 规划依据：
> - 《2024级信管+大数据学年设计II指导手册》：明确主题、功能要求、非功能要求、提交材料和答辩要求。

## 1. 项目目标

本项目以“重庆市二手房源价格数据分析与可视化”为主题，采用“Python 后端 + Android 前端”的方式完成一个可演示、可说明、可提交的系统：

1. 使用 Python 后端完成房源数据采集、清洗、存储、增量更新、统计分析和 API 输出。
2. 使用 Android Kotlin 作为前端 App，调用 Python 后端 API 展示房价概览、区域对比、筛选列表、可视化图表和分析结论。
3. 数据规模目标不少于 5 万条；如果实际网站采集受限，保留“合规采集 + 本地缓存 + CSV/JSON 导入”的兜底方案。
4. 最终按模板补全《学年设计II文档》，正文不少于 8000 字，并包含系统设计、实现过程、关键代码、运行截图和结论。

## 3. 技术路线

### 3.1 工程结构

后端不再放进 Gradle 模块，而是在现有 Android 项目旁边新增 `backend/` 目录：

```text
Houses
├─ app                 Android Kotlin 前端，后期展示数据和图表
├─ backend             Python 后端，接口、爬虫、清洗、入库、分析
│  ├─ app
│  │  ├─ main.py        FastAPI 入口
│  │  ├─ api            接口路由
│  │  ├─ services       业务服务：采集、清洗、统计、分析
│  │  ├─ crawlers       不同数据源爬虫
│  │  ├─ parsers        页面解析
│  │  ├─ db             数据库连接、表结构、仓库
│  │  ├─ schemas        请求/响应模型
│  │  └─ core           配置、日志、异常处理
│  ├─ data
│  │  ├─ raw            原始页面或原始 JSON 缓存
│  │  ├─ processed      清洗后的 CSV/JSON
│  │  └─ houses.db      SQLite 数据库
│  ├─ tests             后端测试
│  ├─ requirements.txt
│  └─ README.md
└─ docs                规划、说明书素材、答辩材料
```

这样分工最清楚：

1. `backend` 负责把数据拿到、处理好、通过接口吐出来。
2. `app` 只负责调用接口和展示，不直接爬网页、不直接读数据库。
3. 文档第 4、5、6 章可以分别对应“采集、清洗存储、分析可视化”。

### 3.2 Python 后端设计

推荐使用 FastAPI。原因是它写接口直观，启动后自动提供 Swagger 接口文档，方便用浏览器测试接口。

后端分层：

| 层 | 目录 | 作用 |
| --- | --- | --- |
| 接口层 | `api` | 定义 HTTP 接口，例如开始采集、查看状态、查询房源 |
| 服务层 | `services` | 编排采集、清洗、入库、统计、算法分析 |
| 爬虫层 | `crawlers` | 面向不同网站实现采集逻辑 |
| 解析层 | `parsers` | 从 HTML/JSON 中提取房源字段 |
| 数据层 | `db` | SQLite 建表、增删改查、聚合统计 |
| 模型层 | `schemas` | 定义接口请求和响应格式 |

建议 API：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/health` | 后端健康检查 |
| `POST` | `/api/crawl/start` | 开始采集，可指定区县、页数、数据源 |
| `GET` | `/api/crawl/status` | 查看采集进度和当前状态 |
| `POST` | `/api/data/clean` | 清洗已采集的原始数据 |
| `GET` | `/api/data/summary` | 查看原始数量、清洗数量、均价、区县数量 |
| `GET` | `/api/houses` | 分页查询清洗后的房源数据 |
| `GET` | `/api/stats/districts` | 各区县均价、数量、总价统计 |
| `GET` | `/api/stats/price-distribution` | 价格区间分布 |
| `GET` | `/api/stats/area-price` | 面积与单价散点数据 |
| `GET` | `/api/analysis/clusters` | KMeans 分群结果 |
| `GET` | `/api/analysis/conclusions` | 可直接展示/写入文档的分析结论 |

接口调用关系：

```text
Android App / 浏览器 / Postman
        |
        v
FastAPI 接口
        |
        v
Service 层
        |
        +--> Crawler 爬数据
        +--> Cleaner 清洗数据
        +--> Repository 读写 SQLite
        +--> Analytics 统计和挖掘
```

### 3.3 第一阶段：先做爬数据

第一阶段目标不是马上做 App，而是先证明数据链路能跑通：

```text
采集网页 -> 解析字段 -> 保存原始数据 -> 清洗标准字段 -> 保存数据库 -> 输出摘要
```

验收方式：

```bash
cd backend
python -m app.cli crawl --source anjuke --district 渝北区 --pages 5
python -m app.cli clean
python -m app.cli summary
```

也可以通过接口验收：

```http
POST /api/crawl/start
POST /api/data/clean
GET  /api/data/summary
GET  /api/houses?page=1&page_size=20
```

第一版不要一上来追求 5 万条，先跑通 100-500 条。字段正确、入库稳定后，再扩大区县、分页和价格段。

### 3.4 数据来源策略

不把系统绑定死在一个网站上。设计统一爬虫接口：

```text
HouseCrawler
├─ AnjukeCrawler       优先尝试公开页面采集
├─ LianjiaCrawler      备选；如果被安全中心拦截，不绕过
├─ LocalHtmlCrawler    读取本地保存网页，保证演示稳定
└─ CsvImportCrawler    CSV 导入兜底，避免后续分析被卡住
```

采集原则：

1. 只访问公开页面，不绕过登录、验证码和封禁。
2. 请求间隔 2-5 秒，避免高频访问。
3. 每页原始内容先缓存到 `backend/data/raw`，便于复现和排错。
4. 每条房源用详情链接哈希或来源 ID 去重。
5. 采集失败要记录原因，不让整个任务崩掉。

### 3.5 数据库与数据字段

MVP 阶段优先使用 SQLite 文件数据库，便于本机演示和提交；如果后续需要，也可以迁移到 MySQL/PostgreSQL。数据库字段先围绕统计和可视化设计：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `source` | 数据来源 |
| `source_id` | 来源站点房源 ID |
| `title` | 房源标题 |
| `district` | 区县 |
| `community` | 小区 |
| `address` | 地址或片区 |
| `total_price_wan` | 总价，单位万元 |
| `unit_price_yuan_m2` | 单价，单位元/平方米 |
| `building_area_m2` | 建筑面积 |
| `rooms` / `halls` | 户型 |
| `floor` | 楼层信息 |
| `orientation` | 朝向 |
| `build_year` | 建成年份 |
| `tags` | 标签 |
| `detail_url_hash` | 详情页 URL 哈希，用于去重 |
| `first_seen_at` | 首次采集时间 |
| `last_seen_at` | 最近采集时间 |
| `batch_id` | 采集批次 |

清洗规则：

1. 去掉空价格、空面积、异常面积、异常单价记录。
2. 统一价格单位：总价转万元，单价转元/平方米。
3. 统一区县名称，如“渝北”“渝北区”归一为“渝北区”。
4. 户型提取为室、厅、卫等数值字段，便于筛选和算法处理。
5. 记录清洗前后的数量，用于文档中说明数据质量。

当前 `app` 模块还是默认页面，后续改造成数据可视化 App。MVP 页面如下：

| 页面 | 展示内容 |
| --- | --- |

## 4. 数据分析与挖掘方案

优先实现能稳定答辩、容易解释的分析方法：

1. 描述性统计：各区县均价、总价中位数、面积分布、房源数量排名。
2. 相关性分析：面积、总价、单价、建成年份之间的关系。
3. KMeans 分群：按单价、面积、总价、建成年份等特征把房源分为若干类，例如“核心高价小户型”“改善型中高总价”“外围低单价大面积”等。
4. 结论生成：后端根据统计结果生成结构化结论，前端展示，文档第 6 章引用。

可选增强：

1. 简单价格预测模型：用面积、区县、户型、年份估算单价或总价。
2. 增量更新对比：展示两次采集之间新增房源和均价变化。
3. 数据质量面板：展示清洗前数量、清洗后数量、去重数量、异常数量。
